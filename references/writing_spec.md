# PRD 写作规范

本规范定义 PRD 文档的章节结构、编号规则、字段表、验收标准、边界异常等专项写作要求。

## 1. 文档体系

采用主文档 + 模块文档两级体系：

| 类型 | 编号格式 | 示例 |
|------|----------|------|
| 写作规范 | SPEC-PRD-00 | SPEC-PRD-00 PRD 写作规范 |
| 全局定义 | PRD-00 | PRD-00 全局角色与权限定义 |
| 模块文档 | PRD-XX | PRD-03 线路管理 |

**需求编号**由需求列表统一管理，格式为 REQ-XXX，在 PRD 中直接引用。

**Use Case 编号**格式：UC-[模块号]-[序号]，例如 PRD-03 中的第一个用例为 UC-03-01。

**文件命名**：`PRD-XX_模块名称_vX.X.md`

## 2. 单份 PRD 标准结构

章节顺序固定，不可调换。无内容时写「暂无」，不得删除章节标题。

```
PRD-XX  模块名称
│
├── 【文档元信息】
│
├── 0. 修订记录
│
├── 1. 项目背景与目标
│     1.1  背景
│     1.2  目标
│
├── 2. 用户分析
│
├── 3. 核心场景
│
├── 4. 业务流程
│
├── 5. 产品架构
│
├── 6. 具体需求
│     6.X  功能名称（REQ-XXX）
│     │     ├── 功能说明
│     │     ├── 交互逻辑
│     │     ├── 字段定义
│     │     ├── 接口/数据边界
│     │     ├── 边界与异常
│     │     └── 验收标准
│
├── 7. 数据埋点
│
├── 8. 风险评估
│
├── 9. 迭代计划
│
└── 10. 附录（术语表、参考资料、修订记录）
```

## 3. 文档元信息

位于文档最顶部：

| 字段 | 说明 |
|------|------|
| 文档编号 | PRD-XX |
| 文档名称 | XX 模块需求文档 |
| 版本 | v1.0 |
| 状态 | 草稿 / 评审中 / 正式发布 / 已废弃 |
| 作者 | 姓名 |
| 最后更新 | YYYY-MM-DD |

状态说明：
- **草稿**：尚未评审，内容可能不完整
- **评审中**：已提交评审，禁止单方面修改
- **正式发布**：评审通过，研发可以依此开发
- **已废弃**：已被新版本替代

## 4. 核心场景（Use Case）

每个 Use Case 描述「某角色在某场景下完成某目标的完整路径」。

标准格式：
```
UC-XX-YY  [角色名称]完成[目标描述]

角色：[角色名称]
场景：[什么情况下触发]
前置条件：
  - 条件一
  - 条件二
入口：[从哪个页面/功能入口进入]

主流程路径：
  1. [角色]进入[页面]
  2. 点击「[按钮名]」
  3. 填写[字段A]（必填）
  4. 点击「确认」，系统校验通过
  5. 跳转回列表页

关联需求：REQ-XXX, REQ-XXX
```

路径变体表格：

| 变体类型 | 触发条件 | 系统响应 | 最终结果 |
|----------|----------|----------|----------|
| 取消操作 | 填写中途点击「取消」 | 弹出确认弹窗 | 关闭弹窗，数据不保存 |
| 校验失败 | 必填字段为空 | 字段红框高亮+报错文案 | 停留在当前页 |
| 无权限 | 非管理员角色访问 | 页面显示无权限提示 | 不展示操作按钮 |

## 5. 功能详述

每个 REQ 对应一节，包含五个子项：

| 子项 | 内容说明 |
|------|----------|
| 交互逻辑 | 界面行为、联动规则、实时校验逻辑 |
| 字段定义 | 字段表格（见第 6 章） |
| 接口/数据边界 | 关键入参出参、权限校验点、与其他模块的数据交互边界 |
| 边界与异常 | 异常场景表格（见第 8 章） |
| 验收标准 | Given-When-Then 格式（见第 7 章） |

## 6. 字段定义表规范

### 6.1 标准格式

| 字段名 | 展示名 | 类型 | 必填 | 限制规则 | 说明 |
|--------|--------|------|------|----------|------|
| phoneNumber | 线路号码 | String | 是 | 11 位数字，以 1 开头 | 唯一，不可重复 |
| carrier | 运营商 | Enum | 是 | CHINA_MOBILE / CHINA_UNICOM / CHINA_TELECOM | 下拉选择 |

### 6.2 列说明

- **字段名**：后端字段名（camelCase）
- **展示名**：前端界面显示的标签文字
- **类型**：String / Integer / Enum / Boolean / DateTime / File
- **必填**：是 / 否 / 条件必填（注明条件）
- **限制规则**：长度限制、格式要求、枚举值等
- **说明**：业务含义、特殊规则、默认值

### 6.3 枚举值写法

Enum 类型必须在限制规则列中穷举所有枚举值，不可写「见代码」或省略。

## 7. 验收标准（Given-When-Then）

### 7.1 格式规范

每个操作至少 1 条（正常路径 + 主要异常路径各 1 条）：

```
场景1：正常新增
  Given  管理员已登录，当前列表页
  When   填写合法数据，点击「确认」
  Then   弹窗关闭，列表首行新增记录，状态为「已启用」

场景2：格式错误
  Given  用户在新增弹窗中
  When   输入非法格式内容，点击「确认」
  Then   字段红框高亮，报错文案「请输入有效内容」，弹窗不关闭

场景3：数据重复
  Given  系统中已存在该数据
  When   再次新增相同数据
  Then   提交后返回错误，提示「该数据已存在，请勿重复添加」
```

### 7.2 写作要点

| 段落 | 描述内容 |
|------|----------|
| Given | 初始状态：谁、在哪、前提是什么 |
| When  | 触发动作：做了什么操作，具体参数是什么 |
| Then  | 期望结果：界面变化、数据变化、提示信息（含具体文案） |

- 每条验收标准对应一个独立场景，不混合多个条件
- 文案类验收标准必须写出具体提示文案，不可写「显示报错」

## 8. 边界与异常场景

### 8.1 标准格式

| 场景编号 | 场景描述 | 触发条件 | 系统响应 | 备注 |
|----------|----------|----------|----------|------|
| E-001 | 停用使用中的资源 | 资源当前被使用中 | 弹出确认弹窗，提示影响范围 | 二次确认后方可操作 |
| E-002 | 删除被依赖的数据 | 数据正被其他功能引用 | 按钮置灰不可点击 | 删除保护逻辑 |
| E-003 | 超出配额限制 | 当前已达数量上限 | 操作按钮置灰，提示配额信息 | - |
| E-004 | 网络异常 | 提交时接口超时或失败 | Toast 提示「操作失败，请稍后重试」 | 通用兜底 |

### 8.2 必须覆盖的通用异常

以下异常在每个涉及写操作的功能中均需覆盖：
- 网络超时 / 接口报错
- 无权限访问（角色校验）
- 数据被其他数据依赖，不可直接删除/停用（依赖保护）
- 并发操作冲突（如两人同时编辑同一条记录）
- 输入校验失败（格式、长度、重复）

## 9. 状态机图规范

### 9.1 适用场景

以下情况必须画状态机图：
- 功能实体有 2 个及以上状态
- 状态之间的流转存在条件限制
- 状态影响其他功能的可用性

### 9.2 绘图语法

使用 Mermaid stateDiagram-v2，需标注：
- 所有状态节点（含初始态和终态）
- 每条流转箭头上的触发动作
- 每个状态下可用的操作（以注释形式标注）

```mermaid
stateDiagram-v2
    [*] --> 草稿 : 新建

    草稿 --> 已发布 : 点击「发布」
    已发布 --> 已停用 : 点击「停用」
    已停用 --> 已发布 : 点击「启用」
    草稿 --> [*] : 删除

    state 草稿 {
        note: 可编辑 / 不可分享 / 不可绑定
    }
    state 已发布 {
        note: 正常服务 / 可分享 / 配置只读
    }
    state 已停用 {
        note: 服务中断 / 已绑定保留 / 可重新启用
    }
```

### 9.3 状态枚举表

状态机图之后，必须附状态枚举表：

| 状态值 | 中文名 | 说明 | 可执行操作 |
|--------|--------|------|------------|
| DRAFT | 草稿 | 新建未发布 | 编辑、发布、删除 |
| PUBLISHED | 已发布 | 正常运行中 | 停用 |
| DISABLED | 已停用 | 已停止服务 | 启用、删除 |

## 10. 写作语言与风格

### 10.1 语言

- 全文使用简体中文
- 技术术语、枚举值保留英文原文（如 ACTIVE、camelCase）
- 界面文案用「」标注，如：点击「确认」按钮

### 10.2 描述规范

| 场景 | 正确写法 | 错误写法 |
|------|----------|----------|
| 描述用户操作 | 用户点击「新增」按钮 | 用户 click 新增按钮 |
| 描述系统响应 | 系统弹出确认弹窗，文案为「……」 | 系统会弹一个弹窗 |
| 描述字段限制 | 不超过 30 个字符 | 不能太长 |
| 描述条件 | 当状态为「已停用」时 | 停用的时候 |
| 描述枚举 | 状态枚举：ACTIVE（已启用）/ INACTIVE（已停用） | 状态有启用和停用 |

### 10.3 禁止事项

- 禁止写「待定」「后续确认」而不标注跟进人和截止日期
- 禁止在正文中写「如研发需要，再补充字段」——字段必须在 PRD 阶段明确
- 禁止用截图代替文字描述——截图可作为辅助，但文字描述必须独立完整
- 禁止在一份 PRD 中定义另一份 PRD 的内容（跨模块逻辑注明「详见 PRD-XX」）

## 11. 非功能要求

描述性能、兼容性、数据保留等非功能性约束。无特殊要求时写「遵循产品通用标准」。

| 类型 | 示例要求 |
|------|----------|
| 响应时间 | 列表查询 ≤ 500ms（P95） |
| 并发 | 支持 50 个并发操作 |
| 数据保留 | 记录保留 180 天，到期后归档 |
| 兼容性 | 支持 Chrome 100+、Edge 100+、Safari 16+ |
| 安全 | 所有写操作须通过服务端权限校验 |

## 12. 数据结构汇总

附录性质，汇总本模块涉及的核心数据对象、字段定义和枚举值，供研发设计数据库和接口时参考。

示例：

| 字段名 | 类型 | 说明 | 枚举值 |
|--------|------|------|--------|
| id | String | 系统自动生成，唯一标识 | - |
| status | Enum | 状态 | ACTIVE / INACTIVE |
| createdAt | DateTime | 创建时间 | - |
